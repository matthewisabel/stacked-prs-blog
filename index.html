Git in the Agentic Era

How AI agents are reshaping version control — and why your branching strategy is about to matter more than ever.

01 The Cambrian Explosion of Commits

Something fundamental shifted when AI agents started writing code. Not the quality of individual lines — that's been debated endlessly — but the sheer volume and velocity of changes flowing into repositories. A single developer paired with an AI agent can produce the commit throughput of a small team. Five developers with agents? You're looking at the commit graph of a mid-size open source project.

This isn't hypothetical. Teams using Copilot coding agents are already seeing branches created, pushed, and PR'd in minutes. The traditional mental model — one developer, one branch, one review cycle — is buckling under the weight of a world where your AI collaborator doesn't need coffee breaks.

The question isn't whether this throughput is coming — it's here. The question is: what does your Git workflow look like when it needs to absorb this?

02 How Agents Think About Branches

Here's a subtle but important insight: AI agents don't "think" about branches the way humans do. A developer builds a mental model of the whole feature, then decides how to decompose it into commits. An agent operates differently — it receives a task, produces a solution, and needs a place to put it. Its branching strategy is a function of its task decomposition, not its understanding of the overall architecture.

This means agents naturally produce work that maps well to stacks: each task is a self-contained unit that depends on the previous one. The challenge is giving agents a framework to express this dependency explicitly, rather than dumping everything into one monolithic branch.

03 The Merge Conflict Problem at Scale

Merge conflicts are annoying for humans. For agents, they're catastrophic. An agent that encounters a conflict can't resolve it with the same intuition a human developer brings — understanding which side "wins" requires the kind of holistic context that agents are still building toward. So instead, we need workflows that minimize the surface area for conflicts in the first place.

This is where stacked PRs become not just a nice workflow, but a structural necessity. When work is organized into a linear chain of dependencies, the merge semantics are unambiguous. Layer 1 merges first. Layer 2 rebases onto the updated main. Layer 3 follows. There's no diamond-shaped merge graph, no ambiguous three-way merge — just a clean, linear history.

04 Stacks as Agent Primitives

With gh stack, agents have a first-class API for expressing their work as ordered, dependent layers. Running gh skill install github/gh-stack teaches an agent how to:

  • Initialize a stack — creating the first branch and establishing the chain
  • Add layers — each new unit of work gets its own branch, automatically targeting the previous layer
  • Cascade rebases — when a lower layer changes, all layers above it rebase automatically
  • Submit the stack — creating PRs for every layer with a single command

This transforms the agent's output from "here's a big diff" into "here's an ordered sequence of reviewable changes." The agent doesn't need to be brilliant about decomposition — it just needs to follow the protocol.

# Agent workflow: from task to stack
gs init auth-model           # Layer 1: data model
# ... agent writes model code ...
gs add  auth-routes          # Layer 2: API routes
# ... agent writes route handlers ...
gs add  auth-middleware       # Layer 3: middleware
# ... agent writes middleware ...
gs add  auth-tests           # Layer 4: test suite
# ... agent writes tests ...

gs push                      # Push all branches
gs submit                    # Create the full stack of PRs

05 Designing for Parallel Agents

The next frontier isn't one agent — it's many agents working simultaneously. Imagine dispatching five agents to work on five different features, each producing its own stack. The repository becomes a coordination problem, and Git is the coordination layer.

Stacked PRs provide natural isolation boundaries. Each stack is a self-contained chain of changes. Two stacks can evolve independently as long as they don't touch the same files. And when they do overlap, the linear merge semantics of stacks make resolution predictable: one stack merges first, the other rebases on top.

🔑 Design Principle

The best Git workflows for agents aren't about controlling the agent — they're about giving the agent mechanical sympathy for how Git works. Stacks provide that sympathy by mapping cleanly to linear dependency chains, which is how agents naturally decompose tasks.

06 The Human-in-the-Loop Review Surface

Here's the part that matters most: agents produce code, but humans review it. And reviewing a 2,000-line PR from an agent is genuinely harder than reviewing a 2,000-line PR from a colleague. You can't interrupt the agent in a hallway to ask "what were you thinking here?" You can't rely on shared context from a planning meeting.

Stacked PRs solve this by making the agent's reasoning visible in the structure of the stack itself. Each layer is a decision: "first I built the data model, then I built the routes on top of it, then I added the tests." The stack is the explanation.

This matters especially for security-sensitive changes. When an agent touches authentication code, you want to review just the authentication layer in isolation. You want CI running against just that layer. You want approval workflows scoped to just that change. Stacks give you all of this for free.

07 What Changes, What Stays

Git itself doesn't change. Commits, branches, merges, rebases — these primitives are as relevant in the agentic era as they were in 2005. What changes is the workflow layer on top of them.

The agentic era demands workflows that are:

  • Programmatic — agents need CLIs and APIs, not GUIs
  • Linear — dependency chains, not dependency graphs
  • Reviewable — small, focused units of change
  • Automatable — cascading rebases, batch merges, CI per layer

Stacked PRs — and specifically gh stack — deliver all four. They aren't a new idea, but they're an idea whose time has come. The agentic era doesn't just make stacked PRs useful; it makes them necessary.

🚀 Getting Started

Install the gh stack CLI extension, teach your agent to use it with gh skill install github/gh-stack, and start breaking those monolith PRs into reviewable stacks. Your future self — and your future agents — will thank you.

01 The Big PR Problem

We've all been there. You open a pull request and the diff is 1,200 lines. Twelve files changed. The description says "Add user authentication and fix some related stuff." Your heart sinks a little.

Research consistently shows that review quality degrades sharply as PR size increases. A 50-line PR gets careful, line-by-line scrutiny. A 500-line PR gets a skim. A 1,200-line PR gets a "LGTM 🚢" and a prayer.

⚠️ The Real Cost

Big PRs don't just get worse reviews — they get slower reviews. Every day a PR sits open is a day its branches diverge further from main, accumulating merge conflicts and blocking downstream work.

02 What Is a Stack, Exactly?

A stack is a series of pull requests in the same repository where each PR targets the branch of the PR below it, forming an ordered chain that lands on your main branch. That's the formal definition. But it's easier to understand visually.

Think of it like building a house: you pour the foundation (layer 1), frame the walls (layer 2), install the wiring (layer 3), and finish the drywall (layer 4). Each layer depends on the one below. You can't install wiring without walls, and you can't frame walls without a foundation. But you can get the foundation inspected while you're framing the walls.

GitHub understands stacks natively. When you create a stack, the PR UI shows a stack map — a navigable list of all layers. Branch protection rules evaluate against the final target branch, not the intermediate ones. CI runs as if each layer were targeting main directly.

03 The Anatomy of a Good Layer

Not all stacks are created equal. A great stack has layers that are:

Property What It Means Example
Focused One logical change per layer "Add User model" — not "Add User model and also refactor the logger"
Buildable Each layer compiles and passes CI on its own Layer 2 shouldn't break tests that Layer 1 introduced
Ordered Dependencies flow downward Routes depend on models, not the other way around
Reviewable ~100-300 lines changed Small enough to review in one sitting, big enough to be meaningful

A worked example

Let's say you're adding a new /users API endpoint with authentication. Here's how you might stack it:

Layer 1 · auth-model       → User model + migration      (~120 lines)
Layer 2 · auth-routes       → CRUD endpoints              (~200 lines)
Layer 3 · auth-middleware    → JWT middleware + guards     (~150 lines)
Layer 4 · auth-tests        → Integration + unit tests    (~280 lines)

Total: 750 lines of code. Four PRs averaging ~190 lines each. Each one reviewable in under 15 minutes. Each one independently testable. Each one a complete thought.

04 How Merging Works

This is where stacks get elegant. When you merge the bottom layer of a stack, GitHub automatically retargets the next layer to point at main. The cascading merge is seamless — you can merge the entire stack in one click, or merge layers individually as they're approved.

05 Rebases: The Cascade

The hardest part of working with stacked branches has always been the rebase problem. Change something in layer 1, and layers 2, 3, and 4 all need to rebase. Manually, this is tedious and error-prone. With gh stack, it's one command: gs rebase. Or one click from the GitHub UI.

Here's what happens under the hood:

The key insight is that cascading rebases are semantically simple even though they're mechanically complex. Each layer's diff stays exactly the same relative to its parent — only the base changes. This means your review comments stay valid, your CI results stay meaningful, and your mental model of the stack stays intact.

06 Stacks in Practice: Real Patterns

After working with stacks extensively at GitHub, some patterns have emerged:

The Feature Stack

The most common pattern. One feature, decomposed into logical layers. The example above (model → routes → middleware → tests) is a feature stack.

The Refactor-Then-Build Stack

Layer 1 refactors existing code to prepare for a new feature. Layer 2+ builds the feature on top of the clean foundation. This is especially powerful because the refactoring can be reviewed and approved before the feature work lands on top of it.

The Migration Stack

For database migrations or API version bumps: Layer 1 adds the new schema/API, Layer 2 migrates callers, Layer 3 removes the old code. Each layer is independently deployable and rollback-friendly.

07 When Not to Stack

Stacks aren't always the answer. Some guidelines for when a single PR is fine:

  • Small changes: If the whole thing is under 200 lines, just ship it as one PR.
  • Truly independent changes: If two changes don't share files or concepts, they're parallel PRs, not a stack.
  • Urgent hotfixes: When production is on fire, skip the ceremony. One PR, one review, one merge.

The question to ask is: "Would a reviewer benefit from seeing these changes in a specific order?" If yes, stack. If no, don't.

08 Getting Started with gh stack

Getting started takes about 60 seconds:

# 1. Install the CLI extension
gh extension install github/gh-stack

# 2. (Optional) Create a shorter alias
gh stack alias

# 3. Start your first stack
gs init my-first-layer
# ... make your changes and commit ...

# 4. Add another layer
gs add my-second-layer
# ... make more changes and commit ...

# 5. Push and create PRs
gs push
gs submit

🎯 The Bottom Line

Stacked PRs aren't about adding process — they're about removing friction. Smaller PRs are faster to review, easier to understand, less likely to conflict, and quicker to merge. Stacks give you small PRs without losing the ability to ship big features. That's the whole pitch.
